- /login/saml redirected to Microsoft's SAML endpoint:
  https://login.microsoftonline.com/c79467b8-b9cf-4b3a-9fba-009a52d544bb/saml2?SAMLRequest=...
- This is AUC's Azure AD tenant (ID: c79467b8-b9cf-4b3a-9fba-009a52d544bb)

SAML FLOW:
1. Browser hits Canvas /login/saml
2. Canvas generates SAMLRequest XML, base64 encodes it
3. Redirects to Microsoft's SAML endpoint with the request
4. Microsoft shows their login page (NOT Canvas's login page)
5. User enters email → Microsoft shows password page
6. User enters password → Microsoft triggers 2FA
7. User approves 2FA on phone
8. Microsoft generates SAMLResponse, POSTs back to Canvas
9. Canvas validates the assertion, creates session, sets cookies
10. Browser lands on Canvas dashboard

SOLUTION:
- Navigate to /login/saml (triggers redirect to Microsoft)
- Wait for Microsoft's login page to load
- Find email input: 'input[type="email"]' or 'input[name="loginfmt"]'
- Fill email, click submit
- Wait for password page (Microsoft uses multi-step login)
- Find password input: 'input[type="password"]' or 'input[name="passwd"]'
- Fill password, click submit
- Wait for 2FA approval (up to 120 seconds)
- Handle "Stay signed in?" prompt (#idBtn_Back or input[value="Yes"])
- Wait for redirect back to instructure.com
- Extract all cookies from browser context

7. BAILEYS — WHATSAPP BLOCKS IT
================================================================================

GOAL: Free unlimited WhatsApp via open-source library

WHAT IS BAILEYS:
- Node.js library by @whiskeysockets
- Connects to WhatsApp via WebSocket protocol (same as WhatsApp Web)
- Doesn't use a browser — direct protocol implementation
- Much lighter than Puppeteer-based solutions

ATTEMPT 1: Latest version (v7.0.0-rc.9)
- npm install @whiskeysockets/baileys@latest
- Error on connection: 405 "Method Not Allowed"
- WhatsApp's server rejected the connection during device registration handshake
- The rc.9 release sends `passive: true` which WhatsApp interprets as passive listener

ATTEMPT 2: Stable version (v6.7.21)
- Same 405 error
- WhatsApp actively blocks connections from datacenter IPs
- Their server checks if the connecting IP is a residential or datacenter IP
- Datacenter IPs (like your local machine or cloud server) get rejected

POTENTIAL FIX (didn't try):
- Use a residential proxy (Smartproxy, Bright Data) to route traffic
- Makes WhatsApp think you're connecting from a home network
- Costs money — defeats the purpose

TECHNICAL DETAIL:
- Baileys uses the Signal Protocol for encryption
- WhatsApp's protocol version is hardcoded in the library
- When WhatsApp updates their protocol, Baileys breaks
- The library must be updated to match WhatsApp's current protocol version
- This is a constant cat-and-mouse game

9. DEPLOYING PUPPETEER/CHROMIUM ON FLY.IO — A SAGA
================================================================================

GOAL: Run whatsapp-web.js on a cloud server (Fly.io)

PROBLEM: Chromium is a full web browser. Running it on a server is hard.

ATTEMPT 1: System Chromium + PUPPETEER_SKIP_DOWNLOAD
- Installed Chromium via apt-get in Dockerfile
- Set PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium
- Set PUPPETEER_SKIP_DOWNLOAD=true (don't download Puppeteer's Chromium)
- Error: "Timed out after 30000ms while waiting for the WS endpoint URL"
- Root cause: System Chromium version didn't match what Puppeteer expected
- Puppeteer communicates with Chrome via DevTools Protocol (WebSocket)
- Different Chrome versions have different protocol versions
- Mismatch = can't communicate = timeout

ATTEMPT 2: System Chromium + whatsapp-web.js 1.26.1-alpha.3
- Tried an older version of whatsapp-web.js that bundles compatible Puppeteer
- QR code generated successfully!
- But sendMessage() crashed: "Evaluation failed" in ExecutionContext
- The older whatsapp-web.js injects JavaScript into WhatsApp Web's page
- System Chromium's V8 engine was too new for the injected code
- JavaScript API differences between Chrome versions broke the evaluation

ATTEMPT 3: webVersionCache remote
- whatsapp-web.js has a webVersionCache option
- Downloads a specific WhatsApp Web version HTML to avoid version mismatches
- Set remotePath to a GitHub-hosted WhatsApp Web version
- Still got "Requesting main frame too early!"
- The issue was Puppeteer <-> Chrome protocol mismatch, not WhatsApp Web version

ATTEMPT 4: puppeteer-core with explicit executablePath
- Installed puppeteer-core (no bundled Chromium)
- Set executablePath to /usr/bin/chromium explicitly
- npm ci failed: package-lock.json didn't match the new package.json
- Even after fixing, same "main frame too early" error

ATTEMPT 5 (SUCCESS): Let Puppeteer download its own Chromium
- Removed PUPPETEER_SKIP_DOWNLOAD
- Removed PUPPETEER_EXECUTABLE_PATH
- Let npm install download the exact Chrome version Puppeteer expects
- Dockerfile: RUN cd whatsapp-bridge && npm install --production
- This downloads ~170MB Chromium binary during Docker build
- Result: QR code works, sendMessage works, everything works!

CHROMIUM LAUNCH FLAGS (required for Docker/server):
--no-sandbox          # Docker runs as root, sandbox conflicts
--disable-setuid-sandbox  # Same reason
--disable-dev-shm-usage   # /dev/shm is only 64MB in Docker, Chrome needs more
--disable-gpu         # No GPU available on server

MEMORY REQUIREMENTS:
- Default Fly.io VM: 256MB → OOM kill
- 512MB → Chromium launches but struggles
- 1GB → Works but slow (sendMessage takes 20-90 seconds)
- 2GB → Reasonable performance (max for shared-cpu-1x on Fly.io)

11. PYTHON PERCENT SIGNS IN HTML TEMPLATE STRINGS
================================================================================

GOAL: Serve HTML pages with CSS from Python strings

PROBLEM:
- CSS uses % in values: border-radius: 50%, width: 100%
- Python's % operator for string formatting interprets % as format specifiers
- If you write body = "width: 100%" % somevar → crash
- Common "fix": escape as %% → "width: 100%%"

OUR MISTAKE:
- We used %% in CSS inside LANDING_HTML and CALLBACK_HTML strings
- But these strings were NOT being formatted with % — just plain assignment
- LANDING_HTML = """...""" (no % formatting applied)
- The %% rendered literally in the browser: "border-radius: 50%%"
- This broke CSS rules — elements didn't render properly
- The phone number input was invisible because its container had broken CSS

FIX:
- Replaced all %% with % since we're not using % string formatting
- Used replace_all=True in the Edit tool to catch all occurrences

12. CHROME EXTENSION — MANIFEST V3 RESTRICTIONS
================================================================================

GOAL: Build Chrome extension to capture Canvas cookies

PROBLEM 1: Inline scripts blocked
- Wrote <script>function doSomething() {...}</script> in popup.html
- Error: "Executing inline script violates Content Security Policy directive 'script-src self'"
- Manifest V3 has strict CSP: only external script files allowed
- No inline <script> tags, no onclick="..." attributes, no javascript: URLs
- Fix: Move all JS to popup.js, reference with <script src="popup.js">

PROBLEM 2: Inline event handlers blocked
- Used <button onclick="save()"> in popup.html
- Same CSP error — onclick is an inline event handler
- Fix: Use document.getElementById('btn').addEventListener('click', fn) in popup.js

PROBLEM 3: Missing storage permission
- Used chrome.storage.local.set() without declaring permission
- Error: "Cannot read properties of undefined (reading 'local')"
- chrome.storage API requires "storage" in manifest.json permissions array
- Fix: Added "storage" to permissions: ["cookies", "storage"]

PROBLEM 4: Missing host permission for server
- Extension called fetch('https://canvas-reminder-auc.fly.dev/api/register-cookies')
- Error: "Failed to fetch"
- Extensions need host_permissions for every external domain they contact
- Fix: Added "https://canvas-reminder-auc.fly.dev/*" to host_permissions

PROBLEM 5: Old extension cached
- After fixing popup.html, Chrome still showed old version
- Chrome aggressively caches extension files
- Clicking "reload" icon in chrome://extensions sometimes isn't enough
- Fix: Completely remove extension, then "Load unpacked" again

13. CANVAS API PAGINATION — URL DOUBLING
================================================================================

GOAL: Fetch all assignments (Canvas paginates results, 10 per page)

HOW CANVAS PAGINATION WORKS:
- Response includes Link header: <https://domain/api/v1/courses/123/assignments?page=2>; rel="next"
- Client follows the "next" URL to get the next page
- Continue until no "next" link

OUR BUG:
- httpx.Client base_url was set to "https://aucegypt.instructure.com/api/v1"
- Our code called _api_get("/courses/123/assignments")
- httpx prepended base_url: "https://aucegypt.instructure.com/api/v1/courses/123/assignments" ✓
- Canvas returned next page URL: "https://aucegypt.instructure.com/api/v1/courses/123/assignments?page=2"
- We parsed the URL to get the path: "/api/v1/courses/123/assignments?page=2"
- httpx prepended base_url again: "https://aucegypt.instructure.com/api/v1/api/v1/courses/123/..."
- Double /api/v1 → 404 Not Found

FIX:
- Changed base_url to just "https://aucegypt.instructure.com" (no /api/v1)
- Added /api/v1 prefix to all path constants in the code
- Pagination URLs from Canvas already include the full path, so no doubling

14. WHATSAPP @LID vs @C.US — IDENTITY FORMAT MISMATCH
================================================================================

GOAL: Receive and respond to all WhatsApp messages correctly

WHAT IS @LID:
- WhatsApp has two identifier formats for users:
  1. @c.us — based on phone number: "201154069714@c.us"
  2. @lid — Linked ID, an internal identifier: "100111610835028@lid"
- LID was introduced with WhatsApp's multi-device architecture
- Some accounts (seemingly random) use LID instead of phone-based IDs
- LID is NOT the phone number — it's an opaque identifier

HOW IT BROKE OUR APP:
- User registers with phone number "201002416119" via Chrome extension
- Database stores: phone = "201002416119"
- User sends "hi" on WhatsApp
- Bridge receives message from "100111610835028@lid"
- Bridge strips @lid → forwards "100111610835028" to Python
- Python looks up "100111610835028" in database → NOT FOUND
- Python sends "You're not registered" message

ALSO BROKE SENDING:
- Reminder tries to send to "201002416119"
- Bridge formats as "201002416119@c.us"
- WhatsApp delivers via @c.us for some accounts, but LID accounts need @lid
- Message sent but never delivered (silently dropped)

FIX (PARTIAL):
- Bridge: strip both @c.us and @lid from incoming messages
- Bridge: detect LID format (>15 digits) and append @lid instead of @c.us when sending
- Database: manually duplicate user entry with LID as phone field
- Used SSH to run SQL on Fly.io server to create LID entries

FIX NEEDED (PERMANENT):
- When user first messages bot, detect if they're using LID
- Auto-update their database entry to use LID as primary identifier
- Store both phone number and LID for reference

16. MULTI-PROCESS IN DOCKER — NODE + PYTHON
================================================================================

GOAL: Run both Node.js (WhatsApp bridge) and Python (FastAPI) in one container

ATTEMPT 1: supervisord
- Installed supervisor via apt-get
- Created supervisord config with two [program:] sections
- Problem: supervisor's stdout_logfile=/dev/stdout caused issues on Fly.io
- Fly.io captures stdout for logging — supervisor's file-based approach conflicted

ATTEMPT 2 (SUCCESS): Simple bash script
  #!/bin/bash
  echo "Starting WhatsApp bridge..."
  node /app/whatsapp-bridge/index.js &    # Background process
  echo "Starting Python server..."
  exec /app/venv/bin/uvicorn src.webhook:app --host 0.0.0.0 --port 8000  # Foreground

KEY DETAILS:
- Node runs in background (&) — if it crashes, it's gone (no auto-restart)
- Python runs in foreground (exec) — if it crashes, container restarts
- Only port 8000 (Python) is exposed to the internet
- Node listens on port 3001 internally (localhost only)
- Python proxies requests to Node when needed (e.g., /qr endpoint)

WHY NOT TWO CONTAINERS:
- Fly.io free tier: limited to minimal resources
- Two containers = double the memory cost
- Communication between containers adds latency
- Single container = localhost communication (fast, simple)

17. NPM CI vs NPM INSTALL IN DOCKER
================================================================================

GOAL: Reliable dependency installation during Docker build

npm ci:
- Requires package-lock.json to exist and exactly match package.json
- Deletes node_modules/ and installs from scratch
- Faster and more reproducible than npm install
- FAILS if package.json was changed without running npm install locally first

npm install:
- Reads package.json, resolves dependencies, generates/updates package-lock.json
- More forgiving — works even if lock file is outdated
- Slightly slower but always works

OUR PROBLEM:
- We changed package.json (new dependencies) without running npm install locally
- Committed package.json without updated package-lock.json
- Dockerfile had: RUN npm ci --production
- npm ci saw the mismatch and failed

FIX:
- Changed Dockerfile to: RUN npm install --production
- Also: removed package-lock.json from .dockerignore so it gets copied if present
- For production apps: always run npm install locally, commit both files, use npm ci in Docker

KEY TECHNICAL ARCHITECTURE DECISIONS
================================================================================

1. COOKIE-BASED AUTH vs TOKEN-BASED AUTH
   - Tokens are better (long-lived, revocable, standard)
   - Cookies were our only option since AUC blocks token creation
   - Cookies expire, need refresh, are tied to browser sessions
   - If we could get an OAuth developer key from AUC IT, everything would be simpler

2. CHROME EXTENSION vs BOOKMARKLET vs SERVER-SIDE PLAYWRIGHT
   - Server-side Playwright: requires user credentials on our server (rejected)
   - Bookmarklet: user drags link to bookmarks, clicks after login (friction)
   - Chrome extension: install once, auto-detects Canvas login (chosen)
   - Tradeoff: extension requires "developer mode" install steps (can't publish to
     Chrome Web Store without $5 developer fee... which requires a credit card)

3. WHATSAPP-WEB.JS vs GREEN API vs BAILEYS
   - Green API: works but costs $5/month
   - Baileys: free but WhatsApp blocks it
   - whatsapp-web.js: free, works, but needs Chromium (memory-heavy)
   - Tradeoff: we use more server memory to avoid monthly costs

4. SINGLE CONTAINER vs MICROSERVICES
   - Could split Node bridge and Python server into separate containers
   - Chose single container for simplicity and localhost communication speed
   - Tradeoff: if bridge crashes, Python can't send messages until next restart

5. SQLITE vs POSTGRES
   - SQLite: zero setup, single file, perfect for <1000 users
   - Postgres: better for concurrent writes, requires separate service
   - Chose SQLite stored on persistent Fly.io volume
   - Will need migration if we exceed ~1000 concurrent users

19. WHATSAPP-WEB.JS ZOMBIE CONNECTION — SILENT MESSAGE LOSS (2026-03-19)
================================================================================

GOAL: Understand why reminders stopped arriving on March 19 after 11am

SYMPTOMS:
- Bridge health endpoint returned {"status":"ok","connected":true}
- /send endpoint returned {"success":true,"id":"3EB0..."}
- But messages never reached users on WhatsApp
- No errors in logs — everything looked healthy
- Reminders appeared to send successfully but were silently dropped

ROOT CAUSE:
- Puppeteer's internal JavaScript context got corrupted
- The underlying Chromium browser process was alive but its DevTools Protocol
  session was broken
- The whatsapp-web.js Client object still had isConnected=true because the
  'disconnected' event never fired
- sendMessage() returned a result object (the library thought it sent) but
  the actual WebSocket to WhatsApp Web was dead
- This is a known issue with Puppeteer on memory-constrained servers (2GB)

HOW WE DISCOVERED IT:
- Sent test message via SSH: flyctl ssh → node -e "fetch('/send'...)"
- Bridge returned success, asked user — message never arrived
- Confirmed: zombie connection — reports healthy but actually dead

WHY IT HAPPENED:
- 2GB Fly.io VM running Chromium (500MB+) + Node + Python
- Memory pressure causes Chromium's V8 engine to become unstable
- WhatsApp Web page inside Chromium may reload or crash silently
- Puppeteer's DevTools Protocol connection breaks but doesn't fire disconnect
- The bridge has no mechanism to detect this zombie state

ATTEMPTED FIX:
- Restarted Fly.io machine: flyctl machines restart
- But Chromium failed to start again: "Runtime.callFunctionOn timed out"
- Then on retry: "The browser is already running" (zombie lock file)
- Container had no kill/pkill/ps commands to manage processes
- Full machine restart needed, then QR re-scan required

DECISION: This was the final straw — replaced whatsapp-web.js entirely

20. REPLACING WHATSAPP-WEB.JS WITH BAILEYS (2026-03-20)
================================================================================

GOAL: Eliminate Chromium dependency, stop zombie connection crashes

OPTIONS EXPLORED:
1. whatsapp-web.js (current) — Puppeteer/Chromium, keeps crashing on 2GB server
2. Baileys (@whiskeysockets/baileys) — WebSocket only, no browser, ~80MB RAM
3. whatsmeow (Go) — Same approach as Baileys but in Go, would need full rewrite
4. WPPConnect — Also Puppeteer-based, same problem as whatsapp-web.js
5. wa-automate — Puppeteer-based + moved to paid model
6. Evolution API — Baileys wrapped in REST API, more overhead
7. Green API — Works but costs $5/month, no credit card

CHOSEN: Baileys

30. BAILEYS "WAITING FOR THIS MESSAGE" — PHONE vs DESKTOP (2026-03-22)
================================================================================

GOAL: Fix messages showing "Waiting for this message. This may take a while."
      on recipient's phone, while the same messages display fine on desktop

SYMPTOMS:
- Bot sends messages via Baileys
- Messages appear correctly on WhatsApp Desktop/Web
- Same messages show "Waiting for this message" on recipient's phone
- Fly.io logs show: "Bad MAC Error" and "No matching sessions found"

ROOT CAUSE:
- WhatsApp phone clients use the Signal Protocol's sender-key mechanism
  differently than linked desktop devices
- When a phone's encryption session gets out of sync, it asks Baileys to
  re-send the message with fresh encryption keys
- Baileys calls getMessage(key) to retrieve the original message content
- WITHOUT a getMessage handler, Baileys returns undefined
- The phone never gets the decrypted content → permanent "Waiting" state
- Desktop clients use a different session type that doesn't need retries

THE FIX — THREE ADDITIONS TO makeWASocket():

1. getMessage handler:
   - When a phone requests message re-encryption, this function provides
     the original message content from an in-memory cache
   - Returns { conversation: '' } as fallback (prevents permanent "waiting")
   - Code: getMessage: async (key) => messageStore[key.id] || { conversation: '' }

2. msgRetryCounterCache (using node-cache):
   - Tracks how many times each message has been retried
   - Without it, retries loop infinitely or fail silently
   - MUST be declared outside the socket (survives reconnects)
   - Code: const msgRetryCounterCache = new NodeCache({ stdTTL: 600 })

3. In-memory message store:
   - Caches all sent and received messages for 10 minutes
   - Sent messages: stored after sock.sendMessage() returns
   - Received messages: stored in messages.upsert handler
   - Auto-cleanup via setTimeout to prevent memory leak
   - Code: messageStore[msgId] = message; setTimeout(() => delete..., 10*60*1000)

4. generateHighQualityLinkPreview: false
   - Disabling this avoids an extra upload step that can interfere with
     message delivery on some devices

ALSO REQUIRED:
- npm install node-cache (new dependency)
- Store sent messages: storeMessage(result.key.id, { conversation: message })
- Store received messages: storeMessage(msg.key.id, msg.message)

REFERENCES:
- GitHub Issue #2297: Messages stuck on "Waiting" on iOS
- GitHub Issue #2115: "waiting for this message" after update
- GitHub Issue #1701: Messages never delivered
- Baileys Official Example (example.ts) includes all three components

KEY LEARNING:
- Baileys official example.ts has getMessage, msgRetryCounterCache, and
  message store — they're not optional, they're required for phone delivery
- Desktop clients are more forgiving because they use different session types
- Testing only on desktop will miss this bug entirely
- The getMessage function is the #1 most important missing piece

31. BAILEYS ZOMBIE DETECTION & AUTO-RECOVERY (2026-03-23)
================================================================================

GOAL: Detect and recover when Baileys silently stops delivering messages

PROBLEM:
- Baileys bridge ran with `&` in the bash start script — if Node crashed,
  it just died silently with no restart
- The /health endpoint only checked `isConnected` boolean, which stayed true
  even when the WebSocket was actually dead (zombie connection)
- Signal Protocol session keys (session-*.json, pre-key-*.json) go stale
  after every deploy/restart, causing "Waiting for this message" on phones
- No proactive monitoring — failures only detected when someone tried to send

THE FIX — 5 LAYERS OF DETECTION:

Layer 1: Auto-clear stale session keys on every startup
- Dockerfile's start.sh runs rm -f on session-*.json, pre-key-*.json,
  sender-key-*.json BEFORE launching Baileys
- Prevents "Waiting for this message" after deploys/restarts
- creds.json and app-state-* files are preserved (no QR re-scan needed)
- Fresh encryption keys are negotiated automatically on first contact

Layer 2: Auto-restart Node bridge on crash
- Replaced bare `node index.js &` with a while-true loop in start.sh
- If Node exits for any reason, it restarts in 5 seconds
- No more silent deaths — crash is logged and recovered from

Layer 3: Liveness check every 2 minutes (Node-side)
- After WhatsApp connects, a setInterval runs every 2 minutes
- Calls sock.fetchStatus() with a 10-second timeout
- If the socket is secretly dead, this throws/timeouts
- 2 consecutive liveness failures → force disconnect + reconnect
- Logs "ZOMBIE DETECTED via liveness check"

Layer 4: Send-time zombie detection (Node-side)
- Before every sock.sendMessage(), checks WebSocket readyState
- If readyState !== 1 (OPEN), rejects immediately with zombie flag
- Tracks consecutive send failures via sendFailCount
- 3 consecutive send failures → auto-reconnect
- Returns {zombie: true} in error response so Python knows

Layer 5: Python health monitor every 5 minutes
- Scheduler calls bridge /health endpoint every 5 minutes
- /health now returns: connected, socketAlive, sendFailCount,
  lastSendSecondsAgo, lastRecvSecondsAgo
- Python logs CRITICAL if bridge is down, WARNING if degraded
- /health endpoint (public) includes full bridge status for debugging

ALSO FIXED:
- whatsapp_service.py _send() detects zombie response from bridge
  and raises ConnectionError instead of silently succeeding
- check_health() and is_healthy() functions added for programmatic checks
- /health endpoint shows both Python scheduler status AND bridge status

KEY LEARNING:
- "Connected" is a lie — the connection.update event only fires on
  explicit disconnects, not on silent socket death
- WebSocket readyState is the ground truth, but Baileys doesn't always
  expose it directly — liveness checks are the reliable fallback
- Session keys MUST be cleared on every restart to prevent encryption
  mismatches. The performance cost (re-negotiating keys) is negligible
  compared to the cost of "Waiting for this message" for all users
- Multi-layer detection is essential: no single check catches all failure
  modes. Liveness catches zombie sockets, send-time catches send failures,
  Python monitoring catches bridge crashes
